IIoT Protocol Compatibility Reference

Which connectivity platforms support which protocols — organized by protocol family. For automation engineers who already know what protocols they are running and want to shortlist platforms fast. Coverage is limited to connectivity middleware vendors; analytics and full-stack platforms are not included.

Support levels
Native Built into the platform — no additional modules, plug-ins, or third-party software required
Plug-in Supported via a separately licensed module, plug-in, or third-party add-on
— Not supported, or requires a full upstream translation layer to achieve connectivity

Protocol support claims are sourced from vendor documentation and reflect the platform's native driver or module library. "Native" means the vendor ships and supports the driver directly — it does not guarantee compatibility with every firmware version or device variant. Verify against your specific device models before purchase. See the protocol compatibility guide for what to test.

MQTT / Sparkplug B / UNS

The transport layer for Unified Namespace architectures. Native support means the platform publishes directly to an MQTT broker using its own built-in client — no separate software required. Plug-in support means a third-party module (such as Cirrus Link for Kepware or Ignition) handles the MQTT publishing. For UNS deployments, the distinction matters: plug-in paths add a dependency and a configuration layer that native paths do not.

Protocol Kepware Ignition Litmus HighByte HiveMQ
MQTT 3.1 / 3.1.1 Plug-in Plug-in Native Native Native
MQTT 5.0 — Plug-in Native Native Native
Sparkplug B — Plug-in Native Native Native
MQTT Broker — — — — Native

Kepware MQTT: requires IoT Gateway plug-in (separately licensed). Ignition MQTT: requires Cirrus Link MQTT Transmission module; Custom Namespace feature enables Sparkplug B alignment with ISA-95 UNS topic hierarchy. HiveMQ is a broker — it does not connect to PLCs; it receives from platforms like Litmus, HighByte, or Ignition.

OPC-UA / OPC-DA

OPC-UA is the current standard for industrial interoperability. OPC-DA is the legacy Windows-only predecessor still running in older SCADA environments. For most new architectures, OPC-UA client support is sufficient — it allows the platform to read from an existing OPC-UA server on the network. OPC-UA server support means the platform itself can expose data to other OPC-UA clients.

OPC-UA Client Native Native Native Native Plug-in

HiveMQ OPC-UA: via the HiveMQ Edge OPC-UA Protocol Adapter extension. Ignition OPC-DA: via Legacy OPC-DA module. Litmus and HighByte consume from OPC-UA servers as clients but do not expose an OPC-UA server interface themselves.

Legacy Rockwell / Allen-Bradley

The Allen-Bradley protocol landscape spans three generations with meaningfully different connectivity requirements. Current-generation Logix platforms (ControlLogix, CompactLogix) use EtherNet/IP. Older SLC 500 and MicroLogix platforms use DH-485 or DF1 serial. PLC-5 platforms use DH+. Each requires different hardware and driver support. See the legacy PLC telemetry guide for hardware requirements.

Protocol / Platform Kepware Ignition Litmus HighByte HiveMQ
EtherNet/IP (Logix) Native Native Native Native —
DF1 Serial (SLC 500) Native Plug-in Native — —
DH-485 (SLC 5/01–5/03) Native Plug-in Native — —
DH+ (PLC-5, SLC 5/04) Native Plug-in Native — —

DH+ and DH-485 require hardware bridging (ProSoft AN-X2, DL-Plus, or NetAIC) before the software driver can connect — "Native" here means native driver support given appropriate hardware. Ignition legacy Rockwell protocols require the Allen-Bradley legacy driver module. HighByte and HiveMQ do not include native Rockwell legacy drivers; data must arrive via OPC-UA or MQTT from an upstream translator.

Siemens

Siemens protocol support splits across three controller generations with different connectivity models. Current S7-1200 and S7-1500 platforms use S7Comm+ with native OPC-UA servers built in. S7-300 and S7-400 use S7Comm via ISO-on-TCP. Older S5 platforms use AS511/TTY and require dedicated hardware gateways. Note the Optimized Block Access issue on S7-1200/1500 — see the protocol compatibility guide.

Protocol / Platform Kepware Ignition Litmus HighByte HiveMQ
S7-1200 / S7-1500 (S7Comm+) Native Native Native Native —
S7-300 / S7-400 (S7Comm) Native Native Native Native —
Profinet Native Plug-in Native — —
S5 (AS511 / TTY) Native — Native — —

S7-1200/1500 Optimized Block Access: platforms that use symbolic browsing via S7Comm+ can read optimized blocks natively. Platforms that rely on absolute ISO-on-TCP addressing require Optimized Block Access to be disabled in TIA Portal. Verify which approach your driver uses before purchase. S5 AS511: requires IBH Link S5++ or equivalent hardware gateway regardless of software platform.

Modbus

Modbus is the most widely deployed industrial protocol by device count — virtually every piece of process equipment from the last 40 years speaks some variant. Modbus RTU runs over RS-232 or RS-485 serial. Modbus TCP runs over Ethernet. Both require careful attention to byte order, word order, and register addressing convention — see the protocol compatibility guide for what to verify before purchase.

Protocol Kepware Ignition Litmus HighByte HiveMQ
Modbus TCP Native Native Native Native Plug-in
Modbus RTU (serial) Native Native Native Native —
Modbus RTU over TCP Native Native Native Native —

HiveMQ Modbus: via HiveMQ Edge Modbus Protocol Adapter extension. Modbus RTU requires a serial-to-Ethernet converter (Moxa NPort or equivalent) before software connectivity is possible — "Native" indicates native driver support given that hardware is in place.

Other Industrial Protocols

Protocols common in energy, utilities, and process industries. DNP3 is standard in electric utilities and water infrastructure. IEC 61850 and IEC 60870 are used in grid automation and substation environments. BACnet is the standard for building automation systems.

Ignition DNP3, IEC 61850, BACnet: available via third-party modules from the Ignition Exchange marketplace. HighByte and HiveMQ are optimized for environments where OT data has already been normalized upstream — they do not include native drivers for these specialized protocols.

Cloud and Enterprise Integration

Outbound integration to cloud data platforms and enterprise systems. These are the upstream destinations for data collected via the protocols above. Native cloud connectors reduce the configuration required to route normalized OT data to analytics platforms.

Kepware cloud integrations: via IoT Gateway plug-in using MQTT or REST. Ignition cloud integrations: via MQTT modules or third-party connectors; Ignition's primary strength is SQL historian and on-premise data routing. HiveMQ data lakehouse: via enterprise connector extensions.

This reference reflects platform capabilities as documented by each vendor. Protocol support depth — firmware version coverage, tag type support, register mapping options — varies significantly within each "Native" entry. Before shortlisting any platform, run the pre-purchase compatibility test described in the protocol compatibility guide.
